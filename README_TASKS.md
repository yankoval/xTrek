# Обработчик очередей (Yandex Message Queue) для системы xTrek

Данный проект (`tasks.py`) представляет собой воркер на базе Celery, который обрабатывает события из S3-хранилища, поступающие через очередь сообщений Yandex Queue. Воркер реализует автоматизацию бизнес-процессов маркировки товаров в системе "Честный Знак".

## Основные концепции

### Типы заданий
- **Реальные задания (префикс `T-`)**: Валидированные задания на производство, созданные на основе входящих документов. Для них требуется печать кодов и физическое нанесение.
- **Виртуальные задания (префикс `V-`)**: Задания для вложений (компонентов) наборов. Коды для них не печатаются, а проходят весь цикл (эмиссия, утилизация, ввод в оборот) программно для обеспечения прослеживаемости состава набора.

### Хранилище (S3)
Процесс управляется перемещением или созданием файлов в определенных папках (префиксах) бакета:
- `Задания/`: Входящие файлы заданий на производство.
- `productionOrders/`: Валидированные и виртуальные задания.
- `equipment-tasks/`: Задания для маркировочного оборудования.
- `equipment-reports/`: Отчеты, полученные от оборудования.
- `emissionOrders/`: Подготовленные заказы на эмиссию.
- `emissionReceipts/`: Квитанции от СУЗ о приеме заказа на эмиссию.
- `emissions/`: Статусы заказов на эмиссию.
- `kodes/`: Выгруженные коды маркировки (КМ).
- `utilisationReceipts/`: Квитанции об отчетах о нанесении (утилизации).
- `introduceReceipts/`: Квитанции о документах ввода в оборот.
- `aggSetReceipts/`: Квитанции о документах агрегации наборов.
- `aggReceipts/`: Квитанции о документах агрегации.

## Бизнес-процессы

### 1. Обработка входящего задания и запуск эмиссии
Процесс начинается при появлении нового JSON-файла в папке `Задания/`.

1.  **Валидация (`logic_create_order`)**:
    - Проверка GTIN, нормализация данных.
    - Проверка в Национальном Каталоге (НК): процесс продолжается только если товар является **набором (SET)**.
    - Создание валидированного задания в `productionOrders/` с префиксом `T-`.
2.  **Задание для оборудования**: Создание задания на агрегацию в `equipment-tasks/`.
3.  **Эмиссия**: Создание заказа на эмиссию в `emissionOrders/`.
4.  **Подписание и отправка (`logic_sign_emission`)**: Извлечение заказа, подписание через сервис подписи и отправка в СУЗ. Квитанция сохраняется в `emissionReceipts/`.
5.  **Мониторинг статуса (`logic_update_emission`)**: Периодическая проверка статуса заказа в СУЗ. Когда статус становится `ACTIVE`, создается файл в `emissions/`.
6.  **Получение кодов (`logic_get_emission_kodes`)**: Загрузка кодов из СУЗ и сохранение в `kodes/`.

### 2. Печать и отчет о нанесении (для наборов)
Когда коды появляются в `kodes/`:

1.  **Генерация PRN (`logic_kodes`)**: Создание файлов для печати на принтерах Amica.
2.  **Утилизация виртуальных кодов**: Если это виртуальное задание (`V-`), автоматически создается, подписывается и отправляется отчет о нанесении (утилизации).
3.  **Триггер агрегации**: Для реальных наборов (`T-`) успешное получение статуса утилизации инициирует проверку готовности к агрегации.

### 3. Обработка отчетов оборудования и виртуальных вложений
Процесс запускается при появлении отчета от оборудования в `equipment-reports/`.

1.  **Проверка отчета (`logic_start_equipment_reports`)**: Валидация данных агрегации.
2.  **Отчет о нанесении для набора**: Создание отчета об утилизации кодов набора на основе данных оборудования, его подписание и отправка.
3.  **Создание виртуальных вложений**:
    - На основе состава набора (из НК) создаются виртуальные задания (`V-`) для каждого компонента.
    - Количество компонентов рассчитывается как `Количество_наборов_из_отчета * Множитель_в_наборе`.
    - Файлы сохраняются в `productionOrders/`, что запускает для них процесс эмиссии (см. Процесс 1).

### 4. Жизненный цикл виртуальных вложений
Виртуальные задания (`V-`) проходят те же этапы эмиссии, что и обычные, но после получения квитанции об утилизации (`utilisationReceipts/`) для них выполняется:

1.  **Ввод в оборот (`logic_utilisationReceipt`)**: Создание и отправка документа "Ввод в оборот" в Честный Знак.
2.  **Контроль статуса ввода (`logic_update_introduce`)**: Ожидание статуса `CHECKED_OK`. После успешного ввода компонента инициируется проверка готовности всего набора к агрегации.

### 5. Агрегация виртуального набора и финальная агрегация
Процесс автоматизирует сборку наборов и транспортную агрегацию после завершения всех подготовительных этапов.

1.  **Проверка готовности (`utils.set_ready_check`)**:
    - Запускается автоматически при получении успешного статуса ввода в оборот любого виртуального компонента или при успешном отчете о нанесении основного набора.
    - Проверяет: тип GTIN (должен быть SET), успех утилизации набора и успех ввода в оборот всех компонентов.
2.  **Агрегация наборов (`trigger_set_aggregation_if_ready`)**:
    - Если проверка `set_ready_check` возвращает `setReady`, воркер последовательно:
        - Генерирует отчет об агрегации наборов (`create_equipment_set_report_from_report`).
        - Создает и отправляет документ агрегации наборов в Честный Знак (`SETS_AGGREGATION`).
3.  **Переход к финальной агрегации (`logic_update_agg_set`)**:
    - При получении статуса `CHECKED_OK` для агрегации наборов (в `aggSetReceipts/`), автоматически запускается процесс стандартной транспортной агрегации (`AGGREGATION_DOCUMENT`).
4.  **Финальная проверка и тегирование (`logic_update_agg`)**:
    - После успешного завершения всей цепочки агрегации (статус `CHECKED_OK` в `aggReceipts/`), выполняется вызов `utils.check_aggregation_report`.
    - Процедура анализирует итоговый отчет и устанавливает на файл в хранилище тег `check` с результатами проверки.

## Технические особенности реализации

### Отложенная обработка и повторы (Retry)
Для взаимодействия с API "Честного Знака" и СУЗ используется механизм `autoretry_for` в Celery.
- Если документ или статус еще не готов, задача выбрасывает `RuntimeError`.
- Настроено большое количество повторов (до 200) с экспоненциальной задержкой, что позволяет "ждать" готовности документов до 12 часов и более.

### Конфигурация
Воркер использует файл конфигурации `suz_worker_config.json` (загружаемый через `xtrek.config_loader`), в котором должны быть определены пути к S3-папкам и параметры подключения к Yandex Message Queue.

### Безопасность
Подписание документов происходит через S3-хранилище ("корзина подписи") с выделенным сервисом подписания. Путь к корзине настраивается через параметр `sign` в конфигурации, что позволяет изолировать закрытые ключи.
